Section 01

The Six-Layer Problem

Good frontend isn't just CSS. Every page must solve six independent problems. Tweaking colors only fixes part of Layer 3.

L1 / Narrative
What is this page saying?
Can a visitor understand who, what, proof, and contact in 30 seconds?

30-second test
L2 / Information Architecture
What modules, in what order?
Does section order support user decision-making, not just field stacking?
Relume
design-consultation
L3 / Visual Direction
What does this look like?
Is there a clear visual stance, or does it look like a generic template?
Variant / Godly
design-shotgun
L4 / Design System
How do we stay consistent?
Are tokens and components unified, or scattered with hardcoded values?
tailwind-design-system
CSS custom properties
L5 / Layout Engineering
Does the page actually work?
Desktop and mobile don't break. Scroll areas are correct. No overflow.

Core Judgment

AI can't design good frontends without direction, references, a design system, and screenshot QA. The problem isn't the model — it's the empty prompt.

Section 02

Nine Skills, Nine Principles

Each skill has an underlying design principle, not just a feature list. Applied correctly, they constrain AI output toward intentional design.

Narrative
interactive-portfolio
Portfolio is opportunity conversion, not resume web-ification. Projects as evidence, not lists.
Principle
30 seconds to know: who, what, strongest proof, how to contact.
Visual
frontend-design
Decide visual stance first, then code. Typography, color, space, motion all serve one concept.
Principle
Editorial, minimal, industrial, luxury, toy, archive, report, console — pick one.
Quality Gate
design-taste-frontend
Hard constraints to counteract AI statistical bias: no purple gradients, no default fonts, no card abuse.
Principle
LLM UI has statistical bias. Use rules to counter-calibrate.
Exploration
design-shotgun
Don't iterate on one wrong direction. Expand search space first with genuinely different directions.
Principle
Change information structure, visual density, typography rhythm — not just button colors.
System
design-consultation
Design isn't one page — it's a reusable set of choices. Template families, brand tone, spacing rhythm.
Principle
Define once, apply everywhere. Token and component contracts, not ad-hoc styling.
System
tailwind-design-system
Design judgment must become tokens and component contracts. Single source of truth for colors, spacing, radius.
Principle
Without tokens, every generated page drifts. CSS custom properties are the contract.
Engineering
shadcn-layouts
CSS layout comes from constraint chains. Height inheritance, scroll containers, grid parent-child relationships.
Principle
h-full needs parent height. Flex scroll children need min-h-0. Grid items need grid parents.
Expression
frontend-css-patterns
Visual language is built from reusable CSS patterns. Font pairings, limited palettes, overlapping, texture, motion rhythm.
Principle
Patterns serve content, not decoration. Every gradient, mask, and animation should have a reason.
Conversion
landing-page-design
The hero must communicate value, evidence, and action in 5 seconds. For acquisition pages only.
Principle
Don't apply landing page patterns to personal sites — it makes them look like sales pages.
Section 03

Find References First,
Then Rebuild

The most important workflow: don't let AI design from scratch. Find similar sites, abstract design language, then rebuild.

01
Define Page Type
Personal site? SaaS landing? Report page? Dashboard? Each needs different methods.
→
02
Find References
Visual direction from Godly/Awwwards. Product flows from Mobbin/Page Flows. SaaS pages from Saaspo.
→
03
Screenshot & Archive
Desktop hero, key sections, mobile hero. No screenshots = no design reference.
→
04
Abstract, Don't Copy
Extract: info order, type feel, color relationships, whitespace density, CTA hierarchy, mobile fold.
05
Generate Brief
Turn reference patterns into rules. Not "copy site X" but "left-aligned hero, single accent color, case study cards".
→
06
Implement
Code from brief + tokens + component rules. Only touch real source files, never build artifacts.
→
07
Visual QA
Playwright screenshots on desktop and mobile. Check text overflow, CTA visibility, scroll, images.
→
08
Ship Review
Check copy, credibility, privacy, exaggeration, brand consistency before going live.

Section 04

Default AI vs. Principled Design

What happens when you prompt "make a pretty page" versus following the six-layer framework.

Default AI Output
× Centered hero with gradient background
× Purple-blue color scheme
× Three equal-width feature cards
× Inter or system font for everything
× Large rounded corners on every card
× "Empower the future" style copy
× Only desktop success state
× Heavy box shadows everywhere
× Every section wrapped in cards
× No loading, empty, or error states
Principled Design
✓ Intentional visual stance (editorial, industrial, etc.)
✓ Limited palette: 2 accents + 3 neutrals
✓ Varied section layouts: grid, timeline, table, list
✓ Three-font system: serif titles + sans body + mono labels
✓ Sharp or minimal corners, structural borders
✓ Specific facts, real metrics, concrete language
✓ Desktop + mobile + edge cases tested
✓ Flat surfaces, no gratuitous shadows
✓ Content breathes — whitespace is deliberate
✓ Loading, empty, error, active states designed

How to use this checklist

Before shipping any AI-generated frontend, run through every item. If a Blocker item fails, don't ship. If a Quality item fails, fix it before the next iteration. This checklist is the minimum bar — it doesn't guarantee great design, but it prevents the most common failures.

Section 07

Prompt Templates

The right prompt structure is the difference between generic output and intentional design.

Reference Research
// Step 1: Never start with "make it pretty"

I need a [page type], target user is [user], core goal is [goal].

Do NOT write code yet. First, search/open 5 high-quality similar sites as references, save screenshots, and analyze:
1. Hero information structure
2. Navigation and CTAs
3. Typography personality
4. Color system
5. Section ordering
6. Card / list / image usage patterns
7. Mobile layout
8. What can be abstracted vs. what must not be copied

Output a new design brief, then design from the brief — don't copy any site directly.
Implementation
// Step 2: Code from the brief, not from vibes

Implement based on the design brief above.

Requirements:
- Use real source files, never build artifacts
- Use existing components and tokens; if none exist, define minimum tokens first
- No default AI style: purple gradients, three-column cards, Inter defaults, card overuse
- All content from given facts — no fabricated clients, data, testimonials, awards
- Screenshot-verify desktop and mobile
- Return file paths, test results, screenshot paths
Full Codex Command
// The complete workflow in one prompt

You are generating a candidate personal website for Offer-Compass.

Do not write code yet. Execute in this order:
1. Use interactive-portfolio method to restructure resume narrative
2. Find 5 high-quality reference sites, save desktop + mobile screenshots
3. Analyze: IA, typography, colors, CTAs, section order, mobile layout
4. Abstract into a new design brief — don't copy any site assets
5. Choose template family, define tokens, components, responsive rules
6. Only modify real source files, never build artifacts
7. Screenshot desktop + mobile with Playwright after implementation
8. Check for: fabricated data, privacy leaks, horizontal scroll, broken images, unclickable CTAs
9. Return file paths, screenshot paths, verification results, remaining risks
